Documenting the code

The documentation you are reading is automatically generated from the code with Doxygen.

Documentation is put in special comments, directly in the code so that you can refer to and update the documentation while you're parsing or editing the code. Special pages, such as this one, can be created as .dox files. This also implies the documentation is version controlled along with the code. Doxygen can then automatically generate HTML files, LaTeX, CHM files or other formats.

Please refer to the Doxygen manual for details about Doxygen tags and syntax.

Doxygen conventions

Doxygen allows different styles to document the code but we'll stick to these conventions, chosen for elegance and simplicity:

  • Doxygen commands and tags are prefixed with an at-sign (@)
  • The first line, until the first dot or blank line, of a Doxygen comment will be used as the brief one-line description (JAVADOC_AUTOBRIEF and QT_AUTOBRIEF are on).

Documentation guidelines

Always precede classes, methods and functions with a Doxygen-style comment. For classes, describe its responsabilities and dependencies. For methods and functions, describe what the function does and describe each individual parameter, as well as the return value. Follow the syntax in this example:

/// Short description in one phrase (in this case: Add two numbers together).
///
/// Optionally, here goes a much more detailed description of the function.
/// This longer description can spawn multiple lines, include code samples, etc.
///
///@param a This is the first floating point number to be added
///@param b And this is the second number
///@return The sum a + b
float myAddFunction(float a, float b)
{
...
}

Documenting C++ code

  • Use the three slashes for C++ Doxygen comments (///). After member documentation is done with ///<.
  • the first sentence will be the short description of what you are documenting. The first sentence is ended by a dot (.) or a blank line.

Here is an example of a well documented header file: shader_desc.h

Documenting Python code

For Python code, a special Doxygen filter, doxypy, is used to circumvent some of the limitations of Doxygen regarding Python documentation. This filter has been modified to allow Epydoc tags as well as regular Doxygen tags, so they're easier to write in Eclipse.

  • Document your functions and classes in docstrings when you can or start with a double hash (##).
  • Doxygen tags are valid in docstrings thanks to the doxypy filter.
  • As in C++ the first phrase will be the short description of what you are documenting.
  • Use double hash comments (##) to documents entities elsewhere in the code.
  • Generally, follow the PEP-257 guidelines.

Special documention pages

Special documentation pages such as the one you are currently reading are used to document features that cannot be attached to a specific portion of the code.

To create one, drop a file with the .dox suffix in doc directory and use this template:

namespace htoa {

/** @page page_label This page's title

The page introductory text.

@section section_label The section title

Your documentation text.

*/
}

Do not forget to then reference your page somewhere else in the documentation, for example in the overview.dox representing the main page, with a subpage reference such as:

/// Please refer to @subpage faq.
