Logging

A simple wrapper class, htoa.log.HaLog, eases the logging and adds a new log level: Trace*.

HaLog uses the standard Arnold logging system and is used thoughout HtoA. HaLog is implemented as a singleton.

Please use HaLog in HtoA over the AiMsgX() raw functions provided by Arnold as it helps getting a consistent look of the log in all modules.

Using HaLog

The recommended way of using HaLog is by importing ha_log as log:

from htoa.log import ha_log as log

You can then log messages at various log levels like this:

log.debug('This is a debug message')
log.info('This is an info message')
log.warning('This is a warning message')
log.error('This is an error message')
log.fatal('This is a fatal message, application will now exit')

Besides being simpler to use and type than their AiMsgXXX() Arnold counterparts, another benefit of using HaLog is that the calling class will automatically get written in the log between square brackets, easing the debug.

Here's a sample log output:

00:00:03 751mb | [htoa.object.geometry] Generating /obj/grid/grid1
00:00:03 751mb | [htoa.node.polymesh] Generating mesh: /obj/grid:<empty>:polygons_0

Trace mode

HaLog also contains a set of trace methods, prefixed with the first letter of the log level (eg. 'e' for "error"):

log.dtrace('This is a trace debug message')
log.itrace('This is a trace info message')
log.wtrace('This is a trace warning message')
log.etrace('This is a trace error message')
log.ftrace('This is a trace fatal message, application will now exit')

These methods will be inactive unless HaLog.enable_trace is set to True. [TRACE] is then added to the log line, here's an example output:

00:00:03 758mb | [htoa.node.node] [TRACE] Parameter "subdiv_adaptive_metric" is set
00:00:03 758mb | [htoa.node.node] [TRACE] Parameter "subdiv_uv_smoothing" is set
00:00:03 758mb WARNING | [htoa.node.node] [TRACE] Parameter "disp_map" is blacklisted
00:00:03 758mb WARNING | [htoa.node.node] [TRACE] Parameter "disp_padding" is blacklisted

The trace methods are intended for obsessive logging of everything being done, and are currently used to monitor or debug parameter passing. Normally, even when debugging, printing every single value being passed can be tedious and lead to unreadable logs and this is why trace mode is off by default.
